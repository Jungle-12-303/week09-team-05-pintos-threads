**전체 구조**
`pintos/tests/threads` 테스트는 크게 세 덩어리입니다.

1. `alarm-*`: timer sleep / wake-up
2. `priority-*`: 우선순위 스케줄러 + semaphore/condition ordering + priority donation
3. `mlfqs-*`: advanced scheduler, `-mlfqs` 모드 전용

가장 중요한 흐름은:

```text
alarm 기본
  → priority ready queue
    → semaphore/condition waiter ordering
      → lock priority donation
        → nested/chain donation
          → MLFQS
```

**1. Alarm 테스트 의존성**
| 테스트 | 의존 기능 | 선행 테스트 |
|---|---|---|
| `alarm-zero` | `timer_sleep(0)` 즉시 반환 | 없음 |
| `alarm-negative` | `timer_sleep(음수)` 즉시 반환/크래시 없음 | 없음 |
| `alarm-single` | sleep list, wake tick 관리 | `alarm-zero`, `alarm-negative` |
| `alarm-multiple` | 반복 sleep, 여러 wake tick 정렬 | `alarm-single` |
| `alarm-simultaneous` | 같은 tick에 여러 thread 깨우기 | `alarm-single` |
| `alarm-priority` | alarm + priority scheduler | `alarm-*`, `priority-preempt`, `priority-sema` 계열 |

`alarm-priority`는 이름은 alarm이지만, 실제로는 **동시에 깨어난 thread들이 priority 순서로 실행되는지**를 봅니다. 그래서 alarm 기본 구현만으로는 부족하고 priority scheduler가 먼저 안정되어야 합니다.

**2. Priority 기본 테스트 의존성**
| 테스트 | 의존 기능 | 선행 테스트 |
|---|---|---|
| `priority-preempt` | 높은 priority thread 생성 시 즉시 선점 | 없음 |
| `priority-change` | priority 낮추면 즉시 yield | `priority-preempt` |
| `priority-fifo` | 같은 priority끼리 FIFO/round-robin 유지 | `priority-preempt` |
| `priority-sema` | semaphore waiters를 priority 순으로 wake | `priority-preempt` |
| `priority-condvar` | condition waiters를 priority 순으로 signal | `priority-sema` |

여기서 핵심 기반은 `priority-preempt`입니다. 이게 안 되면 이후 테스트 출력 순서가 거의 전부 틀어집니다.

**3. Donation 테스트 의존성**
| 테스트 | 의존 기능 | 선행 테스트 |
|---|---|---|
| `priority-donate-one` | 단일 lock donation | priority 기본 |
| `priority-donate-multiple` | 여러 lock에서 여러 donation 관리 | `donate-one` |
| `priority-donate-lower` | base priority와 donated priority 분리 | `donate-one` |
| `priority-donate-multiple2` | lock release 순서가 달라도 donation 유지/회수 | `donate-multiple`, `donate-lower` |
| `priority-donate-nest` | nested donation | `donate-one`, `donate-multiple` |
| `priority-donate-sema` | semaphore block 중 donation 유지 | `priority-sema`, `donate-one` |
| `priority-donate-chain` | 깊은 chain donation | donation 전체 |

Donation 계열은 사실상 다음 상태가 필요합니다.

```text
base_priority
effective_priority
waiting_lock
held_locks
lock별 대기자 priority
```

가장 어려운 건 `priority-donate-chain`이고, 마지막에 보는 게 맞습니다.

**4. MLFQS 테스트 의존성**
MLFQS는 일반 priority donation과 별도입니다. 테스트들이 `ASSERT(thread_mlfqs)`를 걸고 실행되므로 `-mlfqs` 모드에서만 의미가 있습니다.

| 테스트 | 의존 기능 | 선행 테스트 |
|---|---|---|
| `mlfqs-load-1` | `load_avg` 기본 계산 | alarm 기본 |
| `mlfqs-recent-1` | `recent_cpu` 매 tick/매초 계산 | `mlfqs-load-1` |
| `mlfqs-fair-2` | 같은 nice 값의 공정 분배 | `recent_cpu`, priority 재계산 |
| `mlfqs-fair-20` | 여러 thread 공정 분배 | `mlfqs-fair-2` |
| `mlfqs-nice-2` | nice 값 반영 | `mlfqs-fair-2` |
| `mlfqs-nice-10` | 여러 nice 값 반영 | `mlfqs-nice-2` |
| `mlfqs-block` | blocked thread의 recent_cpu decay | `recent_cpu`, lock wake ordering |
| `mlfqs-load-60` | 많은 thread의 load_avg 추세 | `mlfqs-load-1` |
| `mlfqs-load-avg` | 장시간 load_avg 정확도/성능 | MLFQS 전체 |

MLFQS는 timer interrupt에서 너무 무거운 작업을 하면 장기 테스트가 흔들릴 수 있습니다. 특히 `mlfqs-load-avg`는 마지막에 돌리는 게 좋습니다.
